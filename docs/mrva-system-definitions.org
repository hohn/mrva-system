* MRVA documentation: system definitions
  This document defines the MRVA components, non-components, and their relation
  to development, deployment, and documentation.

  In summary:

  1. MRVA core applies existing CodeQL queries to existing CodeQL databases
     using the CodeQL command-line tools, and produces SARIF results.
     It performs no query authoring, result interpretation, or presentation.

  2. The only valid interface to the MRVA core consists of exactly three
     requests, issued via =gh-mrva=:
     - submit
     - status
     - get-results

     No other execution interfaces are supported.

  3. User interfaces are explicitly outside the core and exist solely for
     convenience.
     For demonstration of the core interface (developers) and its operation
     (users), simple Tk-based UIs are provided.
     These interfaces are non-authoritative and replaceable.

  4. CodeQL database provisioning is required by the MRVA core but is not part
     of the core itself.
     The MRVA core depends on the existence of a database provider role, not on
     any specific implementation.

     The database provider interface consists of a single operation:
     =list-dbs=.

     A reference implementation of this interface is provided by the =hepc=
     service.
     This implementation targets open-source CodeQL databases and exists to:
     - demonstrate the provider interface
     - enable development and testing
     - provide a complete, runnable system

     The =hepc= service requires substantial engineering effort but remains
     strictly outside the MRVA core.

     In deployed environments, the reference implementation is expected to be
     replaced by site-specific database providers (for example, internal or
     proprietary repositories) without changes to the MRVA core.

** MRVA Core definition
   MRVA core is the minimal, sufficient system that turns
   : (CodeQL databases × queries) → SARIF results
   Nothing else.

   - The core
     - Accepts references to
       - existing CodeQL databases
       - existing CodeQL queries (packs or individual queries)
     - Executes CodeQL deterministically
     - Produces SARIF output
     - Exposes execution state and results

   - The core does not:
     - edit queries
     - render results
     - interpret findings
     - provide rich UIs
     - manage source code checkouts
     - manage DB construction

   - MRVA core is an execution engine, not an IDE, not an analyst tool.
** MRVA Core Interface, Authoritative API
   - Interface Name: =gh-mrva=. This is the only supported interface to the core.

   - Interface Style
     - Shell-driven
     - Scriptable
     - Stateless per invocation

   - Outside the core, this interface may be used:
     - directly via the shell
     - by user interfaces
     - by automated systems (including AI agents)

   - Command contract:
     Exactly three requests are supported.
     - submit
       + Inputs:
         - DB identifiers
         - Query identifiers
         - Optional execution parameters
       + Output:
         - opaque job ID

     - status
       + Input:
         - job ID
       + Output:
         - state ∈ {queued, running, finished, failed}
         - minimal progress metadata

     - get-results
       + Input:
         - job ID
       + Output:
         + SARIF results
         + CodeQL databases (optional, if explicitly requested)

** MRVA Convenience UI for gh-mrva (Optional, Replaceable)
   - Purpose:  Reduce friction for humans. 

   - Implementations
     - Basic UI (Python / Tk)
       + thin wrapper over gh-mrva
       + submit jobs
       + poll status
       + fetch SARIF
       + minimal result listing

       + Properties:
         - disposable
         - non-authoritative
         - allowed to be ugly

     - Fancy UI (VS Code extension)
       + strictly a gh-mrva client
       + no embedded execution logic
       + no hidden or private APIs

   - Key rule:
     If a VS Code extension requires more privileges or APIs than shell access
     to gh-mrva, the design is incorrect.

** MRVA Core Result Handler (Post-Processing Layer)
   - Purpose:
     Transform raw SARIF into a structured, queryable dataset.

   - Responsibilities:
     + ingest SARIF
     + normalize fields
     + load data into DuckDB
     + preserve references to:
       - repository
       - database
       - query
       - execution metadata

   - Output:
     A read-only analytical store.

   - Dependency rules:
     - MRVA core does not depend on DuckDB
     - DuckDB ingestion depends on MRVA output

** Non-core: Query Editing 
   - Scope: Query editing is not MRVA.

   - Location
     + VS Code
     + Plain filesystem
     + Existing CodeQL tooling

   - Relationship to MRVA:
     + MRVA:
       - consumes queries
       - does not assist with authoring
       - does not validate intent

   - This separation avoids:
     + IDE entanglement
     + schema drift
     + toolchain coupling

** Development
   - Definition: Development is everything that exists to change MRVA.

   - Scope Includes
     + source code
     + build scripts
     + test scaffolding
     + experimental tools
     + refactors
     + prototype UIs
     + temporary glue

   - Properties
     + Mutable
     + Incomplete
     + Allowed to be ugly
     + Allowed to be slow
     + Allowed to break

   - Key Rule: Nothing in development is relied upon at runtime unless explicitly
     promoted to deployment.
    
** Deployment
   - Definition: Deployment is the sealed, operated system.
     This is what users run.

   - Scope includes:
     + MRVA core binaries or containers
     + gh-mrva
     + fixed execution environments (Linux)
     + deterministic configuration

   - Properties:
     + immutable (by policy)
     + reproducible
     + predictable
     + auditable

   - Key rule:
     Deployment artifacts must be buildable from development using a fixed,
     documented procedure.

** Documentation
   Documentation is split along the following axes:
   - system: definitions, structure
   - development: setup, use
   - deployment: setup, use
