* MRVA deployment documentation
** Introduction
  This document describes the /deployment-level/ structure of MRVA.  The goal is to
  explain /what/ is deployed and /how the pieces relate/, without descending into
  developer-level or implementation details.

  We therefore stay strictly at the /container level/.  The internal structure of
  each container is fully specified by its =Dockerfile=; those details are treated
  as authoritative and are not duplicated here.  The complete, reproducible
  assembly process—starting from source—is described in the
  [[*MRVA Build Order][MRVA Build Order]] section.

  The canonical list of containers used in a reference deployment is defined in

  - [[../submodules/mrva-docker/docker-compose-demo.yml]]

  with corresponding Dockerfiles located in

  - [[../submodules/mrva-docker/containers/]]

  Only containers that are part of the MRVA system itself are discussed in this
  document; stock infrastructure containers are treated as external services.

** Container groups

   For clarity, containers are grouped by responsibility:

   1. *Internal containers*: MRVA proper
   2. *Stock containers*: off-the-shelf services used by MRVA
   3. *Externally provided services*: required services that are expected to be
      supplied by the deployment environment (a reference implementation is
      provided for convenience)

   This grouping reflects responsibility boundaries rather than implementation
   details.

** Internal containers: MRVA proper

   The following repositories are submodules of this repository
   ([[../submodules/]]) and together constitute MRVA itself:

   - =gh-mrva=
   - =mrvaserver=
   - =mrvaagent=
   - =mrvacommander=
   - =mrvahepc= (reference implementation only; see below)

   Their roles are as follows:

   - =gh-mrva=
     A user-driven, shell-oriented frontend providing three operations:
     1. submit analysis requests
     2. query request status
     3. download results (and optionally CodeQL databases)

   - =mrvaserver=
     The externally facing service.  It receives requests from =gh-mrva= and
     orchestrates their execution by coordinating the internal services.

   - =mrvaagent=
     The internal job runner.  It performs the actual analyses:
     - consumes jobs
     - retrieves database metadata (and possibly databases) via =mrvahepc=
     - runs =codeql database analyze=
     - returns results via RabbitMQ

   - =mrvacommander=
     Shared library code used by both =mrvaserver= and =mrvaagent=.  It contains no
     standalone service logic.

   - =mrva-docker=
     The container-assembly repository that defines how all MRVA containers are
     built and composed:
     https://github.com/hohn/mrva-docker

** Stock containers (infrastructure services)

   These are unmodified, publicly available containers that provide standard
   infrastructure services within the MRVA deployment:

   - =postgres=
   - =rabbitmq=
   - =minio=

   They are treated as replaceable components, provided they satisfy the expected
   interfaces.

** Externally provided services

   MRVA depends on a small number of services that are *logically required* but are
   expected to be supplied externally in real deployments.  Reference
   implementations are provided solely to make the system self-contained and
   immediately usable.

   - *NOTE*: The default =hepc= configuration references
     https://hohnlab.org/mrva/values/ for access to approximately 70–90 GB of
     open-source CodeQL databases.  For example:

     : result_url = https://hohnlab.org/mrva/values/face-alignment-face-alignment-ctsj-a930b5f031-10d49c933cf269e4.zip

   - This service is *expected to exist* in any MRVA deployment, but its concrete
     implementation is explicitly replaceable.

   - The API is intentionally minimal and stable; see =mrvahepc= for details:
   - =mrvahepc=
     Endpoint for CodeQL database metadata and, optionally, database content.

     The container included with MRVA is provided *only as a reference
     implementation*.  In production deployments, this service is expected to be
     external to MRVA and independently operated.

     Responsibilities:
     - serve CodeQL database metadata within the MRVA network
     - optionally serve CodeQL database content
       (the reference implementation delegates bulk data transfer to an external
       HTTPS server)
     - provide a replaceable, API-stable boundary between MRVA and stored values

     Reference implementation:
     - https://github.com/hohn/mrvahepc/tree/master

*** Fixed API contract of =mrvahepc=

    The =mrvahepc= service exposes a *fixed API* that MRVA relies on.

    - Metadata endpoint:
      - returns a JSON array
      - schema definition:
        https://github.com/hohn/mrvahepc/blob/master/bin/mc-hepc-init#L65
      - example payload:

        #+begin_src javascript
          [
              {
                  "git_branch": "HEAD",
                  "git_commit_id": "dc0c7f3de40530053189c572936ae4fd1567269b",
                  "git_repo": "bulk-builder",
                  "ingestion_datetime_utc": "2022-11-30T00:44:34.455172576Z",
                  "result_url": "http://hepc/db/db-collection.tmp/bulk-builder-bulk-builder-ctsj-a8e72a.zip",
                  "tool_id": "9f2f9642-febb-4435-9204-fb50bbd43de4",
                  "tool_name": "codeql-cpp",
                  "tool_version": "2.11.3",
                  "projname": "bulk-builder/bulk-builder"
              }
          ]
        #+end_src

    - Content serving:
      - actual database content is referenced via the metadata
      - the metadata field name is fixed:
        - ="result_url"=
      - the URL itself is arbitrary and deployment-specific, e.g.:

        ="http://hepc/db/db-collection.tmp/bulk-builder-bulk-builder-ctsj-a8e72a.zip"=

    As long as this API contract is respected, =mrvahepc= can be freely replaced.

** Container assembly

   The assembly and composition of all containers is driven by the
   =mrva-docker= repository.  This deployment document intentionally stops at the
   container boundary; build mechanics and developer-level concerns are covered
   elsewhere.

* MRVA Build Order
  - This document defines execution order.
  - Scripts define state; this document defines sequence.
  - Use a single platform -- amd64 or arm64 -- for all steps to avoid
    cross-platform incompatibilities from Docker.  Docker build and run treat
    =--platform= differently; we don't use it here.
  - MRVA executes natively on a single host architecture per deployment.
    Cross-architecture builds and runtime emulation are intentionally
    unsupported.  Docker’s --platform flag is not used, as it introduces divergent
    semantics between build and run. All artifacts (scratch, images, binaries) are
    host-bound and must not be shared across architectures.

  The following are the individual build steps; they are also in [[../scripts/orchestrate-build.sh][orchestrate-build]].
  1. [[../scripts/prepare-scratch.sh][prepare-scratch.sh]]
  2. [[../scripts/build-mrvaplatform.sh][build-mrvaplatform.sh]]
  3. [[../scripts/build-gh-mrva.sh][build-gh-mrva.sh]]
  4. [[../scripts/build-mrvaserver.sh][build-mrvaserver.sh]]
  5. [[../scripts/build-mrvaagent.sh][build-mrvaagent.sh]]
  6. [[../scripts/build-mrvahepc.sh][build-mrvahepc.sh]]
  7. [[../scripts/build-image-mrvahepc.sh][build-image-mrvahepc]]
  8. [[../scripts/build-image-gh-mrva.sh][build-image-gh-mrva]]
  10. [[../scripts/build-image-mrvaagent.sh][build-image-mrvaagent]]
  11. [[../scripts/build-image-mrvahepc.sh][build-image-mrvahepc]]
  12. [[../scripts/build-image-mrvaserver.sh][build-image-mrvaserver]]

  All scripts:
  - are self-locating
  - write only into deploy-scratch/state
  - emit logs and verification markers
  - may be re-run once prerequisites exist
